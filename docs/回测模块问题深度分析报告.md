## 🔴🔴🔴 致命问题（系统当前无法正常工作）

### 问题 1：超短回测 RPC 调用代码完全错误（🔥 第三轮最严重发现）

#### 事实
`ultra_short.py` 中的 RPC 调用有两处致命错误，导致超短回测的 RPC 通道**完全断路**：

**错误 1：RPCClient 构造函数参数不匹配**
```python
# ultra_short.py 行 209
rpc_client = RPCClient(backtest_host, backtest_port)  # ← 传2个参数

# 但 core/rpc/rpc_manager.py 中 RPCClient.__init__ 签名：
def __init__(self):  # ← 只接受0个参数！→ TypeError!
```

**错误 2：调用了不存在的方法**
```python
# ultra_short.py 行 210
success = await rpc_client.call_submit_ultra_short(task_info)  # ← 方法不存在！

# RPCClient 实际只有：invoke() / broadcast_by_type() / invoke_by_node_id() → AttributeError!
```

**对比：同一项目中单股回测的正确调用方式**（`__init__.py`）：
```python
rpc_client = RPCClient()  # ← 0参数 ✅
results = await rpc_client.broadcast_by_type(
    node_type="backtest",
    method="run_backtest",  # ← 对应 node.py 注册的方法名 ✅
    params=rpc_params,
    timeout=10.0,
    source_node="web-node",
)
```

**node.py 注册的正确方法名**：`run_ultra_short_backtest`，而 ultra_short.py 调用的是 `call_submit_ultra_short`（不存在）。

#### 后果链
1. 用户提交超短回测 → `RPCClient(backtest_host, backtest_port)` 立即抛 `TypeError`
2. except 块捕获 → `mock_tasks[task_id]["status"] = "failed"`
3. **但函数末尾永远返回 `status="running"`**（见问题 2）→ 前端以为任务提交成功
4. 前端轮询 `/status/{task_id}` → 从 mock_tasks 读到 status=failed → 显示"回测失败"
5. **任务从未到达 BacktestNode** → 回测引擎根本没执行

#### 代码拆分背景
原 `backtest.py`（1892行单文件）已拆分为包结构：
- `backtest/__init__.py`（489行）— 单股回测 + 因子选股 + 查询路由 ✅ RPC调用正确
- `backtest/ultra_short.py`（261行）— 超短回测路由 ❌ RPC调用错误
- `backtest/models.py`（231行）— Pydantic 模型
- `backtest/common.py`（40行）— 共享常量
- `backtest/defaults.py`（140行）— 默认配置

拆分时 ultra_short.py 保留了旧版 RPC 调用方式，未与 `__init__.py` 统一。

#### 影响范围
- 🔴🔴🔴 **超短回测 RPC 通道完全断路**，任务无法到达回测引擎
- 🔴 如果回测曾经成功运行过，要么是之前有另一个 RPCClient 实现，要么是回测节点直接从 MongoDB 拉取了任务
- 🔴 旧文件 `backtest.py.original` 仍存在，包含同样的错误代码

### 问题 2：超短回测返回值永远谎报成功（🔥 第三轮发现）

#### 事实
`ultra_short.py` 函数末尾：
```python
# 无论 RPC 成功或失败，都执行到这里
return BacktestTaskResponse(
    task_id=task_id,
    status="running",
    message="回测任务提交成功，正在执行"
)
```

即使 RPC 投递失败（TypeError/AttributeError），except 块只更新了 `mock_tasks`，不抛 HTTPException，函数继续执行到 return，返回 `status="running"`。

**前端行为**：
1. 收到 `status="running"` → 以为任务提交成功
2. 建立 WebSocket 连接或启动轮询
3. 轮询 `/status/{task_id}` → 读到 mock_tasks 中 status="failed" → 显示"回测失败"
4. 用户体验：提交时说成功，几秒后变失败

**对比 `__init__.py` 的正确做法**：RPC 失败时抛 `HTTPException(status_code=503/500)`，前端立即收到错误。

#### 影响范围
- 🔴 用户无法在提交时感知真实错误
- 🔴 前端做了无意义的 WebSocket 连接和轮询，浪费资源
- 🟡 错误信息只能通过日志面板间接看到

### 问题 3：佣金/滑点参数前端传了但后端完全不用（🔥 第三轮发现）

#### 事实
前端 `form.tradeParams` 传了 3 个交易成本参数，但后端全链路没有任何地方使用：

| 参数 | 前端传值 | Pydantic UltraShortParams | node.py | portfolio_backtest.py |
|------|---------|--------------------------|---------|----------------------|
| commission_rate | 0.0003 (万3) | ❌ 无此字段 | ❌ 不读取 | 硬编码 0.0002 (万2) |
| stamp_duty_rate | 0.001 (千1) | ❌ 无此字段 | ❌ 不读取 | 硬编码 0.001 |
| slippage_pct | 0.002 (千2) | ❌ 无此字段 | ❌ 不读取 | ❌ 无滑点模型 |

#### portfolio_backtest.py 硬编码的交易成本
```python
class PortfolioBacktester:
    BUY_COMMISSION = 0.0002     # 买入佣金 万2（前端配的是万3）
    SELL_COMMISSION = 0.0002    # 卖出佣金 万2（前端配的是万3）
    STAMP_TAX = 0.001           # 印花税 千1
    MIN_COMMISSION = 5          # 最低佣金 5元
    # ❌ 完全没有滑点参数
```

#### 影响
- 🔴 前端配置的佣金万3，实际回测用万2 — **交易成本被低估 33%**
- 🔴 前端配置了滑点千2，但回测引擎完全没有滑点模型 — **收益被高估**
- 🔴 用户无法通过界面调整交易成本参数 — 界面配置形同虚设
- 🟡 Pydantic 模型缺少这些字段，即使后端想读取也无处可存

### 问题 4：前端 WebSocket 地址硬编码内网 IP（🔥 第三轮发现）

#### 事实
`UltraShortBacktestViewV2.vue` 行 294：
```javascript
const wsHost = '172.16.16.101:8000'  // 硬编码内网IP
const ws = new WebSocket(`${wsProtocol}//${wsHost}/ws?token=${token}`)
```

#### 影响链
1. 换了网络环境 → WebSocket 连不上 → 降级到 HTTP 轮询
2. 降级轮询只能拿到 `mock_tasks` 中的初始日志（Web API 层生成的 mock_logs）
3. node.py 的 `_push_log` 通过 Redis publish → WebSocket 推送的实时日志在降级模式下**丢失**
4. 前端 `addLog()` 会给每条日志加上本地时间戳 `[HH:MM:SS]`，与后端日志自带的时间戳重复
5. 降级轮询用 `Set` 去重：`[...new Set([...logs.value, ...data.logs])]`，但全量替换会导致数组无限增长

#### 正确做法
```javascript
const wsHost = window.location.host  // 动态获取当前host
```

#### 影响范围
- 🔴 非内网环境无法使用 WebSocket 实时日志
- 🟡 降级模式下日志可能不完整
- 🟡 日志时间戳重复显示

### 问题 5：backtest.py 原始文件代码重复（已拆分，original 仍保留）

#### 事实
原 `backtest.py`（1892行）中存在两套完全重复的路由定义，已拆分为包结构：
- `backtest/__init__.py` — 单股回测 + 因子选股 + 查询路由
- `backtest/ultra_short.py` — 超短回测路由

但旧文件 `backtest.py.original` 仍保留（约1892行），包含：
- 两套重复路由定义
- 两个 `mock_tasks = {}` 字典
- 两个 `BacktestNode()` 实例
- 与 `call_submit_ultra_short` 同样的错误 RPC 调用

#### 事实
`AgentServer/nodes/web/api/backtest.py` 中存在**两套完全重复的路由定义**：

| 第一套（行 264-973） | 第二套（行 1115-1892） |
|---|---|
| `POST /submit` (行 264) | ❌ 不存在 |
| `GET /status/{task_id}` (行 336) | `GET /status/{task_id}` (行 1115) |
| `GET /result/{task_id}` (行 383) | `GET /result/{task_id}` (行 1162) |
| `GET /history` (行 478) | `GET /history` (行 1257) |
| `DELETE /{task_id}` (行 561) | `DELETE /{task_id}` (行 1340) |
| `GET /factors` (行 628) | `GET /factors` (行 1407) |
| `POST /factor-selection` (行 650) | `POST /factor-selection` (行 1429) |
| `POST /ultra-short` (行 750) | `POST /ultra-short` (行 1529) |
| `GET /ultra-short/defaults` (行 974) | `GET /ultra-short/defaults` (行 1753) |

#### 关键差异
- **第一套** `POST /ultra-short`：通过 RPC 投递到独立回测节点（`call_submit_ultra_short`）
- **第二套** `POST /ultra-short`：也通过 RPC 投递，但代码略有差异

**FastAPI 路由注册行为**：后注册的同路径路由会**覆盖**先注册的。因此，**当前实际生效的是第二套代码**（行 1115-1892），第一套代码完全无效（死代码）。

#### ⚠️ 修正之前的分析
之前认为"第一套是 RPC 投递，第二套是进程内执行"——实际两套都是 RPC 投递。但问题更严重：

1. **两套 `mock_tasks` 字典互相独立**：行 743 和行 1522 各定义了一个 `mock_tasks = {}`，由于 Python 变量作用域，第二套的定义覆盖了第一套。如果 status/result 查询在第一套路由生效的瞬间被调用，会读不到数据
2. **`POST /submit`（单股回测）被第二套覆盖后不存在了**：第二套没有 `POST /submit` 路由，**单股回测功能当前完全不可用**
3. **两个 `_backtest_node` 实例**：行 746 和行 1525 各创建了一个 `BacktestNode()` 实例，浪费资源且可能造成初始化冲突

#### 影响范围
- 🔴 **950行死代码**增加维护负担，容易改错地方
- 🔴 单股回测功能不可用
- 🟡 两套代码的 `mock_tasks` 变量各自独立，状态不共享
- 🟡 两个 `BacktestNode()` 实例浪费内存

### 问题 6：默认值三层不一致（第二轮发现）

#### 事实
同一个参数在 Pydantic、前端、node.py 三处定义了**不同的默认值**：

| 参数 | Pydantic (UltraShortParams) | 前端 (Vue) | node.py | portfolio_backtest.py |
|------|---------------------------|-----------|---------|----------------------|
| **stop_loss_pct** | **0.05** (5%) | 0.02 (2%) | 0.02 (2%) | - |
| **take_profit_pct** | **0.1** (10%) | 0.07 (7%) | 0.07 (7%) | - |
| **max_position** | **0.7** (70%) | 0.6 (60%) | 0.6 (60%) | - |
| **max_position_per_stock** | 0.3 | 0.3 | **0.2 / 0.3 自相矛盾** | - |
| **min_volume_ratio** | - | 1.5 | 1.5 / 2.5 | **2.5 硬编码** |

#### 关键细节
1. **node.py 行 749** 传给 PortfolioBacktester 的 `max_position_per_stock` 默认是 **0.2**，但同文件行 575 日志打印默认是 **0.3** — 同一个参数同文件两个值
2. **Pydantic 的 `stop_loss_pct=0.05`** 与前端/node.py 的 **0.02** 完全不一致 — 当前不影响因为前端总是传值，但如果某天前端不传或传空，Pydantic 会用 5% 的默认值，而用户预期是 2%
3. **`portfolio_backtest.py` 行 546 和 565 硬编码了 `min_volume_ratio=2.5`**，完全无视前端提交的 1.5 — **这意味着半路追涨策略的量比阈值实际生效的一直是 2.5，不是用户在界面上设置的 1.5**

#### 影响范围
- 🔴 **portfolio_backtest.py 的硬编码 2.5 导致用户设置无效** — 这是最严重的影响
- 🔴 Pydantic 默认值与实际使用不一致 — 未来潜在 bug 源
- 🟡 node.py 自身内部默认值矛盾

### 问题 7：`volume_threshold` vs `min_volume_ratio` 字段名混乱（第二轮发现）

#### 事实
同一个量比参数，使用了两个不同的字段名，分布在 7 个位置，有 5 个不同的默认值：

| 位置 | 使用的字段名 | 默认值 |
|------|------------|--------|
| 前端 Vue | `min_volume_ratio` (strategyConfigs) | 1.5 |
| Pydantic | `volume_threshold` (UltraShortParams) | 1.5 (ge=1.0) |
| Web API 提交 | 两个字段都有 | - |
| node.py 行 601 | `.get('volume_threshold', .get('min_volume_ratio', 1.5))` | 1.5 |
| node.py 行 715 | `params.get("min_volume_ratio", 2.5)` | **2.5** |
| node.py 行 763 | `s.get("params", {}).get("min_volume_ratio", 1.5)` | 1.5 |
| portfolio_backtest.py 行 546 | `params.get("min_volume_ratio", 2.5)` | **2.5** |
| portfolio_backtest.py 行 565 | `params.get("min_volume_ratio", 1.5)` | 1.5 |

#### 额外问题：语义混淆
Pydantic 中 `volume_threshold` 的描述是"量能放大倍数"（ge=1.0），但前端把 `min_turnover_rate`（换手率门槛，值=3 表示3%）映射给了 `volume_threshold` — **换手率和量能放大倍数是完全不同的概念**，只是数值上恰好都能通过验证。

#### 影响范围
- 🔴 同一个参数名不同，默认值不同，在不同代码路径上可能取到完全不同的值
- 🔴 用户界面设置的量比 1.5，实际在 portfolio_backtest.py 中可能被覆盖为 2.5
- 🟡 换手率/量比概念被混为一谈

### 问题 8：`trade_date` 数据类型不一致（第二轮发现）

#### 事实
MongoDB `stock_daily_ak_full` 集合中的数据存在**两种 `trade_date` 类型**：

| 数据来源 | trade_date 类型 | 示例 |
|---------|----------------|------|
| 原始批量导入 | **int** | `20260106` |
| AKShare 实时下载 | **string** | `"20260106"` |

当前数据库中所有 267,930 条记录都是 `source=ak` 且类型为 `int`，**没有 string 类型的记录**。但代码中的隐患仍然存在：

**node.py 行 1050** — AKShare 下载后插入数据库的代码：
```python
trade_date = row["trade_date"].strftime("%Y%m%d")  # → 生成 STRING "20260106"
record = {"trade_date": trade_date, ...}  # → 插入时是 STRING
```

**node.py 行 1027** — 查询时：
```python
"trade_date": {"$gte": int(start_date), "$lte": int(end_date)}  # → 用 INT 比较
```

**如果 AKShare 实时下载的数据被插入**，trade_date 是 string 类型，用 int 查询会导致：
- `$gte: int(20260106)` 无法匹配 `string("20260106")`（MongoDB 跨类型比较行为不一致）
- 插入的数据在后续查询中可能被遗漏

#### 影响范围
- 🔴 当 MongoDB 无数据触发 AKShare 下载时，新插入的记录类型是 string，后续查询可能找不到
- 🟡 已有的 267,930 条记录类型一致（int），暂时不受影响

## 🟡 架构问题

### 问题 11：日志推送链路复杂且有双写（第二轮深度分析）

#### 完整链路
```
┌─ Web API 层 (backtest.py) ─────────────────────────────┐
│  1. submit_ultra_short_backtest() 生成 mock_logs        │
│     → 参数核对日志 (行 825-905)                         │
│     → 存入 mock_tasks[task_id]["logs"] (内存字典)        │
│     → 存入 MongoDB backtest_tasks.logs (行 915)          │
└──────────────────────────────────────────────────────────┘

┌─ 回测节点层 (node.py) ─────────────────────────────────┐
│  2. _execute_ultra_short_backtest() 输出参数日志        │
│     → _push_log() 写入 (行 545-580)                    │
│     → logger 工具类格式化 (带SEQ序号)                   │
│     → MongoDB $push 追加日志 (行 889)                   │
│     → Redis publish "backtest:logs" (行 895)            │
│     → asyncio.sleep(0) 让出事件循环 (行 907)            │
└──────────────────────────────────────────────────────────┘

┌─ Web 节点层 (web/node.py) ─────────────────────────────┐
│  3. subscribe_backtest_logs() 订阅 Redis               │
│     → pubsub.subscribe("backtest:logs")                 │
│     → ws_manager.broadcast_task_update() → WebSocket    │
│     → 每 100ms 轮询 pubsub.get_message()               │
└──────────────────────────────────────────────────────────┘

┌─ 前端 (Vue) ───────────────────────────────────────────┐
│  4. WebSocket onmessage 接收                           │
│     → data.type === 'log' → addLog(data.log)           │
│     → WebSocket 失败时自动降级为 1秒轮询 HTTP API       │
│        → GET /status/{task_id} → data.logs             │
│        → 用 Set 去重: [...new Set([...logs, ...new])]  │
└──────────────────────────────────────────────────────────┘
```

#### 发现的问题
1. **日志双写**：Web API 层的 mock_logs 和 node.py 的 _push_log 输出几乎相同的参数核对信息
2. **MongoDB 双写**：Web API 层存入完整 mock_logs 数组，node.py 用 `$push` 追加 — 可能导致同一条日志在 MongoDB 中出现两次
3. **前端去重不够精确**：AnsiLogPanel 的 `deduplicateLogs()` 基于"去掉时间戳后内容相同"去重，但 node.py 的日志带 SEQ 序号（如 `[0001]`），Web API 层的不带，**导致内容相同但格式不同，去重失效**
4. **Redis subscribe 轮询间隔 100ms** — 高频日志时可能漏消息
5. **WebSocket 无重连补发机制** — 断开重连后中间日志丢失

### 问题 12：策略参数传递链路 5 层解析（第二轮更深入）

```
第1层：Vue form.reactive 
  → 构建请求体（策略名映射、字段重命名、百分比/小数转换）
  → volume_threshold ← min_turnover_rate (换手率→量比，概念混淆!)

第2层：Pydantic UltraShortBacktestRequest 
  → compatibility_convert root_validator（策略名中→英映射）
  → 默认值与前端/node.py 不一致 (stop_loss=0.05 vs 0.02)

第3层：Web API submit_ultra_short_backtest
  → 从 request 对象提取字段，重新打包为 task_info dict
  → selected_strategies 从 request.params 移到 task_info.params
  → 生成 mock_logs（参数核对日志）

第4层：node.py _execute_ultra_short_backtest
  → params.get("params", {}) → req_params
  → strategies = req_params.get("strategies", [])
  → selected_strategies = req_params.get("selected_strategies", [])
  → volume_threshold / min_volume_ratio 双字段兼容

第5层：portfolio_backtest.py PortfolioBacktester.run
  → config dict 提取参数
  → min_volume_ratio 默认 2.5 (硬编码!)
  → 量比阈值可能被覆盖为 2.5
```

每一层都在做**解包→重新打包**，字段名在每层都可能变化。

### 问题 13：`global datetime` 声明

#### 事实
`node.py` 行 433：`global datetime`  
行 10 导入：`from datetime import datetime`

**为什么写了这个？** 推测是某次调试时 `datetime` 被意外遮蔽，添加 `global datetime` 试图修复。但当前代码中函数内没有任何对 `datetime` 的赋值操作，`global datetime` **没有任何实际效果**。

#### 潜在风险
- 🔴 如果未来有人在函数内写 `datetime = xxx`，`global` 声明会让它**修改模块级绑定**，影响整个文件
- 🔴 违反 Python 最佳实践

- 这在之前的用户反馈中已提到："股票名称没有"
