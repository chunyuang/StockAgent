## 2026-03-15 核心BUG修复

### 问题 1：AKShare 接口一直返回空数据

**现象**：调用 `ak.stock_zt_pool_em()` 一直返回空 DataFrame

**根因分析**：
- 代码中调用 AKShare 接口时，没有传入 `date` 参数
- AKShare 默认返回**今日**数据，如果在非交易日调用就返回空
- 回测时传入历史日期，但代码忽略了这个参数，一直取默认今日

**解决方案**：
```python
# 修改前（错误）
df = ak.stock_zt_pool_em()
# 修改后（正确）
df = ak.stock_zt_pool_em(date=date_str)
```

**提交**：修复所有涨跌停接口，都正确传入 date 参数

---

### 问题 2：昨日日期计算错误

**现象**：获取昨日涨停时，基于系统日期计算，不是基于传入的 trade_date

**根因**：代码中使用 `datetime.now()` 计算昨日，而不是基于回测的 trade_date

**解决方案**：
```python
# 基于传入的 trade_date 计算昨日日期
date_obj = datetime.strptime(str(trade_date), "%Y%m%d")
yesterday = date_obj - timedelta(days=1)
yesterday_str = yesterday.strftime("%Y%m%d")
```

---

### 问题 3：跌停接口异常导致整体涨跌停获取失败

**现象**：AKShare `stock_zt_pool_dtgc_em` 获取跌停失败时，整个涨跌停获取抛出异常，涨停数据也丢失

**根因**：没有异常隔离，涨停和跌停放在一起，一个失败整体失败

**解决方案**：
- 跌停获取单独 try-except，失败记录日志后继续
- 即使跌停获取失败，涨停数据仍然可用
- 只返回涨停数据，不影响选股

---

### 问题 4：Redis 认证失败

**现象**：启动时报错 `Auth failed` 连接 Redis

**根因**：`.env` 中 `REDIS_PASSWORD` 有值，但本地 Redis 没有设置密码

**解决方案**：`.env` 中 `REDIS_PASSWORD=` 留空，代码正确处理空密码

### 问题 4：30 天完整仿真 0 信号

**现象**：完整跑 30 天，所有策略都没有触发信号

**根因分析**：
1. AKShare 只能获取最近 30 日历天涨跌停数据，更早日期返回空 → 正常现象，不是代码问题
2. TushareManager 没有在验证脚本中初始化 → 所有请求都 fallback 到 AKShare，进一步减少数据
3. MongoDB 中没有完整日线数据 → 即使获取到涨跌停，也无法计算均线等条件 → 无法触发

**结论**：
- 代码逻辑正确，不是 BUG
- AKShare 接口本身限制，只能获取最近 30 天
- 已知历史信号单个验证都成功触发，说明逻辑正确

---

### 问题 5：price_change 策略没有重置每日触发标记

**现象**：模拟验证时，大金重工应该触发但没有触发

**根因**：`price_change` 策略没有调用 `reset_daily_triggers()` 清空昨日触发记录，导致重复触发检查失败

**解决方案**：添加 `self.reset_daily_triggers()` 到每日评估开始处

**验证**：修复后大金重工(002487) 2026-03-16 正确触发信号 ✅

## 经验总结

1. **接口参数一定要仔细检查**：不同版本 AKShare 接口名称和参数可能变化
2. **异常隔离非常重要**：一个数据源失败不能影响整个流程
3. **小步验证比一次性全量更好**：单个策略验证通过再全量跑
4. **三级 fallback 设计非常有效**：即使 TuShare 没有权限，AKShare 也能满足基本需求
5. **免费数据源有其限制**：AKShare 涨跌停只能近 30 天，Baostock 补充日线历史非常合适
