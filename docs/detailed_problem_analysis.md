## 🧪 核心问题分析

### 1. 数据单位变化的影响

**修复前（数据单位：元）**：
- 平安银行成交额: 1,003,479,224.05 元
- 流动性门槛: 1000万元 = 10,000,000元
- 很多股票满足门槛

**修复后（数据单位：万元）**：
- 平安银行成交额: 100,347.92 万元
- 流动性门槛: 1000万元 (保持不变)
- 仍然应该满足门槛

**结论**: 数据单位转换不是主要问题。

### 2. 数据结构差异

检查修复前后的数据结构对比：

```
修复前 (20260326 - 使用 Tushare 数据源)
- 数据集合: stock_daily (Tushare 格式)
- 字段包括: ts_code, trade_date, open, high, low, close, vol, amount等

修复后 (20260329 - 使用 AKShare 数据源)
- 数据集合: stock_daily_ak (AKShare 格式)
- 字段包括: ts_code, trade_date, open, high, low, close, vol, amount, source, up_limit, down_limit
```

**关键差异**：
1. **数据源标记**: 修复后有 `source="ak_historical"` 字段
2. **字段完整性**: 修复后有完整的 `up_limit`, `down_limit` 字段
3. **数据集合名称**: 从 `stock_daily` 改为 `stock_daily_ak`

### 3. 策略因子引擎兼容性问题

#### 潜在问题1：数据源过滤
- 策略因子引擎可能内置了数据源过滤条件
- 可能只查找 `source=null` 或 `source="ts"` 的数据
- 修复后的数据有 `source="ak_historical"` 可能被排除

#### 潜在问题2：字段命名差异
- 某些因子可能依赖特定命名字段
- AKShare数据字段名可能与Tushare不同

#### 潜在问题3：数据集合切换
- 策略配置中可能硬编码了数据集合名
- 从 `stock_daily` 切换到 `stock_daily_ak` 可能不兼容

### 4. 因子计算逻辑分析

从3月26日的成功回测结果看：
- **半路追涨**: 依赖 `limit_up_yesterday`, `open_below_limit`, `volume_increase` 因子
- **涨停开板**: 依赖 `limit_up_yesterday`, `first_limit_up` 因子

这些因子可能对数据格式有特定要求：
1. **limit_up_yesterday**: 需要前日涨停价数据
2. **open_below_limit**: 需要涨停价和开盘价对比
3. **volume_increase**: 需要成交量对比计算
4. **first_limit_up_: 需要首板涨停识别

## 🎯 问题根源假设

基于对比分析，**最可能的问题根源**：

### **假设1：数据源过滤条件**
- 策略因子引擎可能内置过滤：`source IS NULL OR source = 'ts'`
- 修复后数据：`source = 'ak_historical'`
- **结果**: 所有数据被排除

### **假设2：必需字段缺失**
- 某些因子需要 `pre_close` 或 `change_pct` 字段
- 这些字段在AKShare数据中可能不存在
- **结果**: 因子计算失败，返回空列表

### **假设3：数据格式转换问题**
- 数据类型不匹配（如字符串 vs 浮点数）
- 数值范围异常（如成交额单位问题）
- **结果**: 因子计算错误
