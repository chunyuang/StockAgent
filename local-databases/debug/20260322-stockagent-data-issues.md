## 问题汇总 & 解决方法

### 问题 1: `trade_date` 字段存储为字符串，范围查询失败

**现象:**
- 数据同步时，`trade_date` 一直存储为字符串
- 使用 `{'$gte': 20251217, '$lte': 20260317}` 整数范围查询返回 0 条记录
- 即使后来转换为 int，因为唯一索引冲突导致索引重建失败，查询依然失败

**根因:**
- 并行同步脚本写入时，没有确保类型一致，全部写入字符串
- 去重重建唯一索引时，之前的错误导致索引损坏

**解决方法:**
1. 删除损坏的唯一索引
2. 逐个转换所有 `trade_date` → int 类型
3. 重建唯一索引 `(ts_code ASC, trade_date DESC)` 保证唯一性
4. 单独建立 `trade_date` 索引加速范围查询

**结果:** ✅ 解决，范围查询正常工作

---

### 问题 2: 缺失 `is_limit_up` / `is_limit_down` / `limit_up_price` / `limit_down_price` 字段

**现象:**
- 数据同步只同步了基础价格字段 (`open/high/low/close/pre_close/vol/amount/pct_chg/change`)
- 没有计算涨跌停标记字段，所有这些字段缺失
- 导致所有依赖涨跌停的因子计算都得到 0，选不出股票

**根因:**
- 同步脚本只同步了基础数据，涨跌停标记需要基于 `pct_chg` 计算，同步脚本没有做这一步

**解决方法:**
1. 批量遍历所有记录
2. 基于 `pct_chg` 计算涨跌停:
   - `pct_chg >= 9.5` → `is_limit_up = True`
   - `pct_chg <= -9.5` → `is_limit_down = True`
   - 计算 `limit_up_price = pre_close * 1.1`，`limit_down_price = pre_close * 0.9`
3. 添加这些字段到所有记录

**结果:** ✅ 解决，涨跌停标记正确

### 问题 3: 数值字段存储为字符串，计算错误

**现象:**
- `open/high/low/close/pre_close/pct_chg/change/amount/vol` 全部存储为字符串
- 进行数值比较和计算时出错

**根因:**
- 数据源读取返回字符串，插入MongoDB时没有转换

**解决方法:**
- 批量遍历所有记录
- 将所有数值字段转换为 float
- 总共 **2,552,675** 条记录全部转换完成

**结果:** ✅ 解决，所有数值计算正常

### 问题 4: 并行同步产生重复记录，导致唯一索引创建失败

**现象:**
- 并行同步时，同一个 `(ts_code, trade_date)` 可能被多次写入
- 创建唯一索引时，发现重复键，索引创建失败

**根因:**
- 并行分片同步，没有做去重，产生重复记录

**解决方法:**
- 去重：对于重复的 `(ts_code, trade_date)`，保留一条，删除其他重复
- 删除重复后重建唯一索引

**结果:** ✅ 解决，唯一索引创建成功

### 问题 5: 回测汇总时 `KeyError: 'final_value'` 崩溃

**现象:**
- 回测每个策略都正常完成，但是汇总结果时崩溃
- 错误: `KeyError: 'final_value'`

**根因:**
- 当整个回测区间 **0 交易** 时，`performance` 字典为空
- 汇总代码直接访问 `perf['final_value']` 没有检查，导致崩溃

**解决方法:**
- 修改汇总代码，当 `performance` 为空时，使用 `final_cash` 计算最终值
- 最终资金 = `final_cash` + sum(持仓市值)，0 交易时持仓为空，所以最终资金 = 初始资金 → 正确

**结果:** ✅ 解决，汇总正常完成

**统计验证:**
```

跌停<20 (可交易): 32 天 → 56.1%
```

**结论:**
- 不是 bug，是正确结果
- 这段时间市场整体弱势，接近一半日期跌停超过 20 家，触发强制空仓
- 剩下 32 天可交易，但严格选股条件+因子打分后，依然选不出满足条件的股票 → 所以 0 交易
