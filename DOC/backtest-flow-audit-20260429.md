## 📌 阶段1: 前端提交

**文件:** `frontend/src/views/backtest/` (Vue组件)

**动作:**
1. 用户在界面配置参数（策略选择、日期范围、全局参数、策略独立参数）
2. 点击"开始回测"
3. 前端组装 `UltraShortBacktestRequest` 发送 `POST /api/v1/backtest/ultra-short`

**请求数据结构:**
```json
{
  "strategies": ["halfway_chase", "first_limit_up", ...],
  "start_date": "20260105",
  "end_date": "20260320",
  "initial_cash": 1000000,
  "params": {
    "liquidity_threshold": 500,
    "stop_loss_pct": 0.02,
    "take_profit_pct": 0.07,
    "max_hold_days": 3,
    "max_position_per_stock": 0.2,
    "max_position": 0.7,
    "commission_rate": 0.0003,
    "stamp_duty_rate": 0.001,
    "slippage_pct": 0.002,
    "selected_strategies": [
      {"id": "halfway_chase", "name": "半路追涨", "params": {"min_volume_ratio": 1.5, ...}}
    ]
  },
  "enable_force_empty": true,
  "enable_sentiment_cycle": true,
  "enable_auction_filter": true
}
```

**⚠️ 对齐检查点:**
- [ ] 前端所有可配参数是否都包含在请求中？
- [ ] `selected_strategies[].params` 的字段名是否与后端策略读取一致？
- [ ] 日期格式是否统一为 `YYYYMMDD`？

## 📌 阶段2: API层接收

**文件:** `AgentServer/nodes/web/api/backtest/ultra_short.py`

**动作:**
1. 解析请求体 → `UltraShortBacktestRequest` 验证
2. 生成 `task_id = "us_{uuid}"`
3. 构建 `task_info` (含params、selected_strategies)
4. 写入 MongoDB `backtest_tasks` 集合
5. 写入 `mock_tasks` 内存缓存
6. 通过 `RPCClient.broadcast_by_type()` 投递到 BacktestNode
7. 返回 `{task_id, status: "running"}`

**⚠️ 对齐检查点:**
- [ ] `task_info` 中的参数是否100%透传前端提交值（无遗漏/无覆盖）？
- [ ] `DateTimeEncoder` 序列化是否正确处理了所有 datetime/ObjectId？
- [ ] RPC 超时60s是否足够？大数据量时是否超时？

**🔴 已发现问题:**
- `convert_datetime_to_str()` 递归转换 + `DateTimeEncoder` 是双重防护，但逻辑冗余
- `mock_tasks` 和 MongoDB 双写，查询时优先级不明确

## 📌 阶段3: RPC投递到BacktestNode

**文件:** `AgentServer/nodes/backtest_engine/node.py`

**动作:**
1. `_handle_run_ultra_short_backtest()` 接收 RPC 请求
2. 将任务加入 `_task_queue` (asyncio.Queue)
3. Worker 从队列取出任务
4. 调用 `execute_ultra_short_backtest(task_info, self._push_log, self.logger, task_id)`

**⚠️ 对齐检查点:**
- [ ] 队列是否真的只允许1个并发？(MAX_CONCURRENT=1)
- [ ] Worker崩溃后任务是否丢失？(无持久化队列)

- [ ] **策略筛选**: 每个策略的条件是否与参数配置对齐？
- [ ] **候选合并**: 多策略候选去重后，买入顺序/权重如何分配？
- [ ] **调仓执行**: 买入数量是否遵守100股整数倍+单票仓位限制？
- [ ] **滑点计算**: 使用固定0.2%还是动态滑点？(PortfolioBacktest用固定滑点)
- [ ] **佣金计算**: 是否使用 config 传入的 commission_rate？

**🔴 已发现的关键问题:**

1. **代码重复 (约850~990行)**: `universe_raw`/`universe` 获取和流动性过滤代码出现了**3次**，显然是merge冲突或复制粘贴bug，导致同一交易日重复查询3次MongoDB

2. **非调仓日处理**: 代码中 `if trade_date in rebalance_set` 分支处理调仓日，但**非调仓日的else分支**需要检查是否正确计算了持仓市值变化

3. **净值/回撤计算**: 在非调仓日，如果持仓不变，日盈亏应该基于收盘价变化计算，而非0

## 🔴 全链路关键风险点汇总

| # | 环节 | 风险 | 严重度 | 说明 |
|---|------|------|--------|------|
| 1 | 5.2 | **代码重复** — universe获取+流动性过滤出现3次 | 🔴 P0 | 同一天重复查询3次MongoDB，选股结果可能被覆盖 |
| 2 | 5.2 | **非调仓日净值=0** — 持仓不变时日盈亏为0 | 🔴 P0 | 导致夏普比率=0、回撤不准 |

| 5 | 5.2 | **滑点模型不一致** — PortfolioBacktest用固定0.2%，backtester.py用动态滑点 | 🟠 P1 | 同一策略不同引擎结果不同 |
