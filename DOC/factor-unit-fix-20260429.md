## 🔴 单位不匹配问题汇总

### 问题1: `limit_down_open_amount` 单位不一致 (P0)

| 项目 | 值 | 单位 |
|------|-----|------|
| 数据库存储 | 9.9871 | **万元** |
| 同类因子 `limit_up_open_amount` | 48126.6 | **千元** |
| 同类因子 `limit_up_amount` | 4582.2 | **千元** |
| 筛选参数 `min_qiao_amount` | 10000 | **万元**(界面配置) |

**结论:** `limit_down_open_amount` 被除以10000存入数据库，而其他金额因子（limit_up_open_amount, limit_up_amount）保留了amount原值(千元)。

**影响:** 跌停翘板策略 `limit_down_open_amount >= 10000` 永远不满足（max=9.99）。

**根因推测:** compute_factors.py中 `limit_down_open_amount = x["amount"]` 应该存千元，但可能被后续另一个脚本/流程覆盖为万元值。或者计算时读取的DataFrame中amount已经是万元(而不是千元)。

### 问题2: `min_qiao_amount` 参数单位假设不一致 (P0)

| 策略 | 筛选代码 | 参数值 | 因子实际单位 | 匹配? |
|------|---------|--------|------------|-------|
| 跌停翘板 | `limit_down_open_amount >= min_qiao_amount` | 10000 | **万元**(0~10) | ❌ 差1000倍 |
| 首板打板 | `limit_up_open_amount >= min_seal_amount` | 5000 | **千元**(0~99627) | ⚠️ 5000千元=500万，合理 |

**修复方案:** 
- 如果修正 `limit_down_open_amount` 为千元(与limit_up_open_amount对齐)，则 `min_qiao_amount=10000` = 10000千元=1000万，合理
- 如果保持 `limit_down_open_amount` 为万元，则 `min_qiao_amount` 应改为10(10万元=1000千元=100万)... 不对，这更不对

**正确方案:** 修正 `limit_down_open_amount` 存储为千元(与同类因子一致)，然后 `min_qiao_amount=10000` = 10000千元 = 1000万元 ≈ 1亿元。这仍然偏高，但至少能匹配上。

实际上，10000千元=1000万≈1亿 对翘板来说确实偏高。界面配置写的是"10000万元=1亿"，但实际想表达的可能是1000万？

## 🔧 修复方案

### 修复1: 修正 limit_down_open_amount 存储单位 (P0)

**方案A (推荐): 重新计算并更新数据库中的 limit_down_open_amount**

将 `limit_down_open_amount` 从万元修正为千元(=amount)，与 limit_up_open_amount 一致。

```python
# 修复脚本: update limit_down_open_amount = amount * 10000 (万元→千元)
db.stock_daily_ak_full.update_many(
    {"limit_down_open_amount": {"$gt": 0, "$lt": 100}},  # 只修复明显异常的小值
    [{"$set": {"limit_down_open_amount": {"$multiply": ["$limit_down_open_amount", 10000]}}}]
)
```

**方案B: 修正筛选参数，适配万元单位**

在 `_build_strategy_filter_conditions` 中，将 `min_qiao_amount` 乘以10000，从万元转为千元与因子对齐。

但这依赖 `limit_down_open_amount` 的单位始终是万元，不可靠。

**我选方案A** — 修正数据，从根源解决。
